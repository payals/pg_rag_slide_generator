# Rendering

Jinja2 templates composed from DB-driven fragments, producing self-contained HTML presentations with a custom reveal.js-style engine.

## Slide Types

Six slide types, each with its own content structure and HTML fragment:

| Type | Visual description | `content_data` shape |
|------|-------------------|---------------------|
| `bullets` | Title + bullet list | `{ "bullets": [...] }` |
| `statement` | Large centered statement with optional subtitle | `{ "statement": "...", "subtitle": "..." }` |
| `split` | Two-column layout | `{ "left_column": {...}, "right_column": {...} }` |
| `flow` | Step-by-step pipeline/process | `{ "steps": [...] }` |
| `code` | Code block with language tag and explanation | `{ "language": "...", "code": "...", "explanation": "..." }` |
| `diagram` | Diagram definition with description | `{ "diagram": "...", "description": "..." }` |

## Static Slides

Not generated by the LLM ŌĆö loaded from Postgres at startup:

| Source table | Content |
|-------------|---------|
| `static_slide` | Title slide (intent: `title`) and thanks slide (intent: `thanks`). Contains `title`, `content_data`, and `speaker_notes`. |
| `section_divider` | 5 dividers inserted after specific intents. Each has `title`, `subtitle`, and optional `image_path`. Keyed by `after_intent` FK to `intent_type_map`. |

`inject_static_slides()` assembles the final deck: title slide first, section dividers after their designated intents, thanks slide last, then renumbers all slides sequentially.

### Templates

| Template | Purpose |
|----------|---------|
| `reveal_base.html` | Static HTML export ŌĆö full deck with embedded CSS/JS |
| `reveal_live.html` | Live server deck with SSE connection and status panel |
| `slide_fragment.html` | Single-slide HTML fragment for SSE injection (includes `_slide_type_body.html`) |
| `_slide_type_body.html` | Shared partial with if/elif dispatch for all 6 slide types + image block |

## Fragment Composition

Each slide type's HTML lives in `slide_type_config.html_fragment` (loaded from DB). At startup, `compose_slide_type_body()` assembles them into a complete `_slide_type_body.html`:

The fragment order is derived dynamically from the `slide_type_config` table (all types except `bullets`, sorted alphabetically). The composition builds a Jinja2 if/elif chain in this order, with `bullets` as the `{% else %}` fallback. An image block is appended after the type dispatch.

### Dev Workflow

Fragment source files live in `templates/fragments/`:

```
templates/fragments/
  bullets.html
  code.html
  diagram.html
  flow.html
  split.html
  statement.html
```

`src/load_fragments.py` loads these into `slide_type_config.html_fragment` in Postgres. Use `--check` mode for CI parity verification (ensure DB fragments match filesystem source).

If `compose_slide_type_body()` returns `None` (any fragment missing), the renderer falls back to the filesystem `_slide_type_body.html`.

## Themes

Two themes loaded from the `theme` table:

| Theme | Description |
|-------|------------|
| `dark` | Default dark theme |
| `postgres` | Postgres-branded theme with custom colors |

Themes are applied via `css_overrides` ŌĆö a block of CSS custom properties injected into the rendered HTML. Select with `--theme dark` or `--theme postgres`.

## SSE Streaming

The live server (`src/server.py`) uses two event channels that feed into a unified SSE stream:

### Channel 1: Postgres LISTEN/NOTIFY

| Postgres channel | Trigger function | SSE event type |
|-----------------|-----------------|---------------|
| `slide_committed` | `notify_slide_committed()` (fires on slide INSERT/UPDATE) | `slide_added` |
| `gate_update` | `notify_gate_update()` (fires on gate_log INSERT) | `gate_update` |

### Channel 2: In-process asyncio.Queue

The orchestrator pushes progress events to an `asyncio.Queue`:

| SSE event type | Content |
|---------------|---------|
| `progress` | Phase, current intent, cost estimate, slide count |
| `error_event` | Generation error (named `error_event` to avoid the SSE reserved word `error`) |
| `complete` | Generation finished |

### Browser Integration

5 named SSE event types consumed by the browser JS:

| Event | Browser action |
|-------|---------------|
| `slide_added` | Inject HTML fragment into slide deck |
| `progress` | Update status panel (phase, intent, cost) |
| `gate_update` | Update gate status indicators |
| `error_event` | Display error in status panel |
| `complete` | Show completion state, stop polling |

**Note:** `deck_saved` is emitted when the deck is auto-saved to `output/` but has no client-side listener (dead event).

## Output Format

Static rendering produces a self-contained HTML file:

- All CSS embedded inline (no external stylesheets)
- All JS embedded inline (~200 lines: navigation, speaker notes, progress bar, viewport scaling)
- No dependency on the full reveal.js library ŌĆö this is a purpose-built minimal implementation
- Supports keyboard navigation (arrows, space), speaker notes toggle (S key), and responsive viewport scaling
- Suitable for offline use (no network requests)

Rendered via:

```bash
python -m src.renderer --deck-id <uuid> --theme postgres --output output/deck.html
```
