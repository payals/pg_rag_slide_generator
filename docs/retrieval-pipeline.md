# Retrieval Pipeline

From markdown ingestion through two-stage reranking to LLM context assembly.

## Content Ingestion

### Source Types

Documents are typed via the `doc_type` enum: `note`, `article`, `concept`, `blog`, `external`, `image`. Each document has a `trust_level` (high, medium, low) that influences retrieval ranking.

### Sourcing Tiers

| Tier | Trust Level | Examples |
|------|------------|---------|
| Own content | high | Original notes, project docs |
| Official docs | high | Postgres documentation, extension docs |
| Quality blogs | medium | Vetted technical blog posts |
| External articles | low | Third-party content, general references |

### G0 Ingestion Gate

Documents are validated at ingest time before chunking:
- Title is required and non-empty
- Content meets minimum length threshold
- Content hash is checked for deduplication (skip if already ingested)

## Chunking Strategy

| Parameter | Value | Config key |
|-----------|-------|---------|
| Target chunk size | 700 tokens | `chunk_size_tokens` |
| Overlap | 100 tokens | `chunk_overlap_tokens` |
| Minimum chunk size | 50 tokens | `min_chunk_size_tokens` |
| Tokenizer | tiktoken `cl100k_base` | — |

### Splitting Priority

Chunks are split at the highest-priority boundary available:

1. Markdown headers (`#`, `##`, etc.)
2. Double newlines (paragraph breaks)
3. Single newlines
4. Sentence boundaries

### Metadata Preservation

Each chunk retains: `doc_id` (FK to source document), `section_header` (nearest preceding heading), `content_hash` (for deduplication), and `token_count`.

## Embedding Model

| Parameter | Value |
|-----------|-------|
| Model | OpenAI `text-embedding-3-small` |
| Dimensions | 1536 |
| Storage | pgvector `VECTOR(1536)` column on `chunk.embedding` |
| Cost | $0.00002 per 1K tokens |

Embeddings are generated at ingest time and stored alongside the chunk. Image captions also get embeddings in `image_asset.caption_embedding`.

## Hybrid Search (RRF)

`fn_hybrid_search` combines two retrieval signals via Reciprocal Rank Fusion:

| Signal | Method | Weight (default) |
|--------|--------|-----------------|
| Semantic | pgvector cosine distance (`<=>`) | 0.7 (`semantic_weight`) |
| Lexical | tsvector full-text search (`@@`) with `ts_rank_cd` | 0.3 (`lexical_weight`) |

The function accepts optional filters: `doc_type`, `trust_level`, `tags`. Results include a `combined_score` used for downstream ranking.

### RRF Formula

Each signal produces a ranked list. For each chunk, the RRF score is:

```
rrf_score = semantic_weight / (k + semantic_rank) + lexical_weight / (k + lexical_rank)
```

where `k = 60` (standard RRF constant).

## Coverage-Enriched Retrieval

The orchestrator's `retrieve` node enriches search queries using `v_deck_coverage`. When generating a slide for intent X, the query includes context about what intents are already covered, prompting the retrieval to "differentiate from existing slides." This is the macro/mini pattern: macro context (deck-level coverage) informs micro decisions (per-slide retrieval).

## Two-Stage Reranking

| Stage | Method | Output |
|-------|--------|--------|
| 1. Postgres | `fn_hybrid_search` returns `rerank_top_k` (default 50) candidates | Coarse ranking by RRF score |
| 2. Cross-encoder | `cross-encoder/ms-marco-MiniLM-L6-v2` scores each candidate | Fine ranking, return top `default_top_k` (default 10) |

Reranking is toggled via `rerank_enabled` (default `true` in config table). When disabled, Stage 1 results are returned directly with `top_k` truncation.

The cross-encoder scores each (query, chunk) pair independently, producing a relevance score that's more accurate than embedding cosine similarity for short passages.

## G1 Retrieval Quality Gate

After retrieval, the orchestrator calls `mcp_check_retrieval_quality`, which invokes `fn_check_retrieval_quality` in Postgres. Thresholds are configurable via the `config` table:

- **Minimum chunks:** >= `g1_min_chunks` (default 3) chunks returned
- **Minimum score:** > `g1_min_score` (default 0.3) `combined_score` on the top result

If G1 fails, the orchestrator retries with alternative queries generated by the LLM. See [gate-validation.md](gate-validation.md) for the full gate sequence.

## Image Retrieval

`fn_search_images` provides semantic image search:

- Query embedding compared against `image_asset.caption_embedding`
- Optional filters by `style` and `use_cases`
- Results exclude images already used in the current deck (`used_image_ids`)
- Intent-based boosting: images whose `use_cases` match the current intent get score bonuses

Image selection is controlled by `image_selection_enabled` (default `true` in config table) and score thresholds `image_min_score` (0.5) and `image_intent_min_score` (0.35).
